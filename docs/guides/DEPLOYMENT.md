# Deployment Guide

The canonical guide for deploying Wolf Goat Pig: backend to **Render**, frontend
to **Vercel**. Both auto-deploy on push to `main`. For diagnosing a failed or
misbehaving deploy, see [deployment-troubleshooting.md](./deployment-troubleshooting.md).

## Architecture

```
┌─────────────────┐      ┌─────────────────┐
│   Vercel        │      │   Render        │
│   (Frontend)    │ ───► │   (Backend)     │
│   React SPA     │      │   FastAPI       │
│   static build  │      │   PostgreSQL    │
└─────────────────┘      └─────────────────┘
```

- **Backend**: FastAPI on Render, configured by `render.yaml` (`rootDir: backend`,
  `startCommand: python render-startup.py`, health check at `/ready`). Managed
  PostgreSQL. Database migrations are applied automatically at startup by
  `render-startup.py` (see `app/migrations_runner.py`).
- **Frontend**: React SPA on Vercel, configured by the root `vercel.json`.
- **Auth**: Auth0 (same tenant for both, audiences must match — see below).

## Pre-deploy gate (run before every push)

Pushing to `main` deploys to production. The local gate must mirror CI exactly.
This is the authoritative checklist from the project `CLAUDE.md`:

```bash
# Frontend changes (mirrors .github/workflows/frontend-ci.yml):
cd frontend && npm run typecheck && npx vitest run && npm run build

# Backend changes (mirrors .github/workflows/backend-ci.yml):
cd backend && ruff check app/ tests/ && ruff format --check app/ tests/ \
  && pytest tests/ --ignore=tests/manual --ignore=tests/_diagnostic
```

`typecheck` is **not** optional — esbuild/vite tolerate syntax `tsc` rejects, so a
passing build alone has shipped broken code before.

## Environment variables

> ⚠️ React variables (`REACT_APP_*`) are **baked into the build at compile time**.
> They must be set in Vercel **before** building, not at runtime.
>
> ⚠️ The live Render service does **not** sync env vars from `render.yaml`. Set
> backend env vars in the **Render dashboard** (Environment tab) and verify via
> `/health`.

### Vercel (Frontend) — required

Settings → Environment Variables, for all environments (Production, Preview, Development):

| Variable | Value | Notes |
|----------|-------|-------|
| `REACT_APP_API_URL` | `https://wolf-goat-pig.onrender.com` | Backend API URL — **critical** |
| `REACT_APP_AUTH0_DOMAIN` | `dev-jm88n088hpt7oe48.us.auth0.com` | Auth0 tenant domain |
| `REACT_APP_AUTH0_CLIENT_ID` | `qAZuRv5E9mPQ9uTGg7NWpkpfVj8bCeoB` | Auth0 application client ID |
| `REACT_APP_AUTH0_AUDIENCE` | `https://api.wolf-goat-pig.com` | Auth0 API audience |
| `REACT_APP_USE_MOCK_AUTH` | `false` | **Must be `false` in production** |
| `REACT_APP_ENABLE_ANALYTICS` | `true` | Optional |

Locally the frontend uses the `package.json` proxy (`"proxy": "http://localhost:8000"`),
so `REACT_APP_API_URL` is empty and requests are proxied to the local backend.

### Keeping Render and Vercel in sync

Both platforms read secrets from their own dashboards. To avoid drift, keep a
single `.env.production` source of truth in a secure vault (start from
`config/.env.production.template`) and push the same values to each host. Re-sync
whenever you rotate a secret — missing/mismatched values are the most common
cause of Auth0 failures.

## Deploying

### Backend (Render)

1. Connect the GitHub repo to Render as a Blueprint instance using `render.yaml`.
2. Set `AUTH0_CLIENT_SECRET` (and any other secrets) in the dashboard — not in `render.yaml`.
3. Render builds from `backend/`, runs migrations on startup, and serves via `render-startup.py`.
4. Subsequent deploys happen automatically on push to `main`.

### Frontend (Vercel)

1. Connect the GitHub repo to Vercel.
2. **Before the first deploy**, set all required `REACT_APP_*` variables in the dashboard.
3. Vercel builds with the root `vercel.json` and deploys static assets.
4. Subsequent deploys happen automatically on push to `main`; preview deploys are created for PRs.

## Auth0 configuration

Production auth failures are usually mismatched callback URLs or audiences:

1. **Callback / Logout URLs** — add the Vercel production URL and any preview/custom
   domains to Auth0 → Application → *Allowed Callback URLs* and *Allowed Logout URLs*.
2. **Web Origins** — add the same URLs to *Allowed Web Origins*; ensure the backend's
   CORS origins include the Render and Vercel hostnames.
3. **API Audience** — the Auth0 API Identifier must equal both `AUTH0_API_AUDIENCE`
   (backend) and `REACT_APP_AUTH0_AUDIENCE` (frontend). Tokens for any other audience
   fail backend validation.

## Cold starts (Render free tier)

Render's free tier sleeps after ~15 minutes of inactivity, causing a 30–60s delay
on the first request. The frontend handles this gracefully via `ColdStartHandler`
and `utils/api.js` (exponential backoff with a user-facing wait message), so first
visits show a friendly delay rather than appearing broken. Subsequent visits are
fast while the backend stays warm.

## Post-deploy verification

After a backend deploy:

```bash
# environment must read back "production"
curl https://wolf-goat-pig.onrender.com/health

# a junk Bearer token must return 401, not 500
curl -H "Authorization: Bearer junk" https://wolf-goat-pig.onrender.com/players/me
```

Then confirm:

- Frontend loads at `https://wolf-goat-pig.vercel.app` with no console errors about missing env vars.
- API calls reach `https://wolf-goat-pig.onrender.com`.
- Auth0 login works end-to-end.

Optional scripted validation (run from the repo root):

```bash
python scripts/deployment/verify-deployments.py \
  --backend https://wolf-goat-pig.onrender.com \
  --frontend https://wolf-goat-pig.vercel.app
```

## Rollback

- **Render** — Dashboard → service → *Manual Deploy* → select a previous commit, or
  use the Rollback action.
- **Vercel** — Dashboard → Deployments → *Promote* a previous working deployment.
- **Either** — revert the offending commit and push:
  ```bash
  git revert HEAD && git push origin main
  ```
